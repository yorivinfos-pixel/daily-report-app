1.1 LE PROBLÈME À RÉSOUDRE

Dans le secteur des télécommunications en RDC, des dizaines de sites (tours, 
pylônes) sont construits simultanément à travers le pays. Chaque site passe 
par 17 phases de construction (de l'implantation au nettoyage final).

Problèmes avant l'application :
- Les superviseurs terrain envoyaient des rapports par WhatsApp ou email
- Pas de format standardisé → données incomplètes, incohérentes
- Les PM passaient des heures à compiler les rapports manuellement
- Impossible de suivre l'avancement en temps réel
- Pas de photos organisées (perdues dans les fils WhatsApp)
- Aucun système de scoring ou d'alerte de retard

Solution : une application web accessible depuis le terrain (téléphone) qui 
centralise, standardise et automatise tout le processus de reporting.

1.2 LES ACTEURS DU SYSTÈME

Nous avons identifié 5 rôles avec des besoins différents :

Superviseur terrain
  - Besoin : soumettre facilement des rapports journaliers avec photos
  - Contrainte : réseau internet faible/intermittent, utilise un smartphone
  - Interface : simple, formulaire rapide, adapté mobile

Project Manager (PM)
  - Besoin : voir tous les rapports de sa zone, donner des feedbacks
  - Contrainte : analyse sur ordinateur, export Excel/PDF
  - Interface : dashboard complet avec filtres et statistiques

Program Manager
  - Besoin : supervision de 2 zones, vue globale
  - Interface : même dashboard PM avec accès multi-zones

Group PM
  - Besoin : vue globale toutes zones
  - Interface : dashboard PM sans restriction de zone

Administrateur
  - Besoin : créer/gérer les comptes utilisateurs
  - Interface : panneau d'administration simple

1.3 LES FONCTIONNALITÉS ATTENDUES

Fonctionnalités essentielles (MVP — Minimum Viable Product) :
  ✓ Authentification sécurisée par rôle
  ✓ Formulaire de rapport avec 17 phases
  ✓ Upload de photos (jusqu'à 10 par rapport)
  ✓ Dashboard PM avec filtres et vue d'ensemble
  ✓ Attribution de sites aux superviseurs
  ✓ Export Excel et PDF
  ✓ Gestion des comptes utilisateurs (admin)

Fonctionnalités avancées :
  ✓ Notifications en temps réel (Socket.IO)
  ✓ Chat de zone (superviseurs ↔ PM)
  ✓ Calcul automatique du score superviseur
  ✓ Suivi de tracking par site (work status)
  ✓ Système RFI (Request for Inspection)
  ✓ Document d'acceptation final
  ✓ Application mobile Android (APK)
  ✓ Application bureau (Electron)
  ✓ Système de rebranding (marque blanche)
  ✓ API de zone mapping dynamique

1.4 CHOIX DE L'ARCHITECTURE

Architecture choisie : Application Web Monolithique + API REST

Pourquoi ce choix :
  - Un seul serveur = simple à déployer et maintenir
  - Node.js = même langage (JavaScript) partout (serveur + client)
  - API REST = standard universel, facile à comprendre
  - MongoDB = flexible, pas besoin de schéma fixe au début
  - Pas de framework frontend complexe = plus rapide à développer

Alternatives considérées et rejetées :
  - React/Vue/Angular : trop complexe pour l'équipe, temps de développement
  - Microservices : sur-ingénierie pour cette taille de projet
  - PHP/Laravel : équipe plus à l'aise avec JavaScript
  - Firebase : coûts imprévisibles à grande échelle

2.4 POURQUOI CES CHOIX ?

Express.js plutôt que Fastify/Koa/Hapi :
  → Le plus populaire, la plus grande communauté, documentation abondante
  → Simple à apprendre et à enseigner

MongoDB plutôt que PostgreSQL/MySQL :
  → Schéma flexible (pas besoin de ALTER TABLE quand on ajoute un champ)
  → Documents JSON = naturel pour JavaScript
  → MongoDB Atlas offre un plan gratuit généreux
  → Adapté aux données semi-structurées (rapports avec champs variables)

Vanilla JS plutôt que React/Vue/Angular :
  → Pas de build pipeline complexe (pas de webpack/vite)
  → Déploiement instantané (modifier un fichier = c'est en ligne)
  → Courbe d'apprentissage plus douce pour l'équipe
  → Moins de dépendances = moins de vulnérabilités
  → Inconvénient : code plus verbeux, pas de composants réutilisables

Socket.IO plutôt que WebSocket natif :
  → Reconnexion automatique si connexion perdue
  → Système de "rooms" pour cibler les messages
  → Fallback automatique si WebSocket non disponible

Cloudinary plutôt que AWS S3/Google Cloud Storage :
  → Plan gratuit suffisant pour démarrer
  → Transformation d'images intégrée (redimensionnement, compression)
  → SDK JavaScript simple à utiliser
  → Pas besoin de configurer des buckets/permissions complexes

Render plutôt que Heroku/Railway/DigitalOcean :
  → Plan gratuit avec HTTPS automatique
  → Déploiement automatique depuis GitHub
  → Interface simple, pas de DevOps complexe
  → ⚠️ Inconvénient : système de fichiers éphémère (d'où Cloudinary pour les images)

3.2 LE SERVEUR EXPRESS

Le fichier server.js est le cœur de l'application (~1730 lignes).

Structure générale de server.js :

  1. Imports et configuration (lignes 1-25)
     → require('express'), require('mongoose'), etc.
     → Chargement des variables d'environnement

  2. Connexion MongoDB (lignes 26-40)
     → mongoose.connect(MONGODB_URI)

  3. Définition des Schémas/Modèles (lignes 43-210)
     → User, Report, Site, ChatMessage

  4. Configuration Express et Middlewares (lignes 224-290)
     → helmet, cors, express.json, mongoSanitize, static files

  5. Fonctions utilitaires (lignes 130-290)
     → normalizeProvince, getZoneFromRegion, generateToken, authMiddleware

  6. Routes API (lignes 290-1700)
     → Toutes les routes organisées par domaine

  7. Socket.IO (lignes 1700-1750)
     → Configuration des événements temps réel

  8. Démarrage du serveur (lignes 1710-1730)
     → server.listen(PORT)

Exemple de route API simple :

  // Récupérer tous les sites
  app.get('/api/sites', authMiddleware, async (req, res) => {
    try {
      const sites = await Site.find({});
      res.json({ success: true, sites });
    } catch (err) {
      res.status(500).json({ success: false, error: err.message });
    }
  });

Décomposition :
  - app.get('/api/sites', ...) : quand quelqu'un fait GET /api/sites
  - authMiddleware : vérifie le token JWT (rejeté si pas connecté)
  - async (req, res) : fonction asynchrone avec requête et réponse
  - Site.find({}) : cherche tous les documents dans la collection "sites"
  - res.json({...}) : renvoie la réponse en JSON

3.4 LES MODÈLES DE DONNÉES (SCHÉMAS MONGOOSE)

Mongoose permet de définir la structure attendue des documents.

Exemple — Le modèle User :

  const userSchema = new mongoose.Schema({
    full_name: { type: String, required: true },
    username: { type: String, required: true, unique: true, lowercase: true },
    password_hash: { type: String, required: true },
    role: { 
      type: String, 
      enum: ['admin', 'group_pm', 'program_manager', 'pm', 'supervisor'],
      required: true 
    },
    zone: { type: String, default: '' },
    is_active: { type: Boolean, default: true },
    created_at: { type: Date, default: Date.now }
  });

  const User = mongoose.model('User', userSchema);

Explication :
  - required: true → le champ est obligatoire
  - unique: true → pas de doublon possible
  - lowercase: true → converti automatiquement en minuscules
  - enum: [...] → valeurs autorisées uniquement
  - default: → valeur par défaut si non fournie

3.5 L'AUTHENTIFICATION (JWT + BCRYPT)

Processus de connexion en 7 étapes :

  1. Client envoie : POST /api/auth/login { username, password }
  
  2. Serveur cherche l'utilisateur :
     const user = await User.findOne({ username: 'evariste.famba' });
  
  3. Serveur vérifie le mot de passe :
     const valid = await bcrypt.compare(password, user.password_hash);
     // bcrypt compare le mot de passe saisi avec le hash stocké
  
  4. Si valide, serveur génère un JWT :
     const token = jwt.sign(
       { id: user._id, username: user.username, role: 'supervisor', zone: 'Zone 2' },
       process.env.JWT_SECRET,
       { expiresIn: '7d' }
     );
  
  5. Serveur renvoie : { success: true, token: "eyJhbG...", user: {...} }
  
  6. Client stocke le token : localStorage.setItem('authToken', token)
  
  7. Pour chaque requête suivante, client envoie :
     fetch('/api/reports', {
       headers: { 'Authorization': 'Bearer eyJhbG...' }
     })

Côté serveur, le middleware vérifie le token :

  function authMiddleware(req, res, next) {
    const header = req.headers.authorization;
    if (!header) return res.status(401).json({ error: 'Token manquant' });
    
    const token = header.split(' ')[1]; // Extraire après "Bearer "
    const decoded = jwt.verify(token, JWT_SECRET); // Vérifier + décoder
    req.user = decoded; // Attacher les infos utilisateur à la requête
    next(); // Continuer vers la route
  }

3.7 LA SÉCURITÉ

Couches de sécurité implémentées :

Couche 1 — HTTPS (Render fournit automatiquement un certificat SSL)
  → Toutes les communications sont chiffrées

Couche 2 — Helmet (en-têtes HTTP sécurisés)
  app.use(helmet({ contentSecurityPolicy: false }));
  → Protection contre XSS, clickjacking, MIME sniffing

Couche 3 — CORS (restriction des origines)
  → Seuls les domaines autorisés peuvent appeler l'API
  → Les apps mobiles et localhost sont toujours autorisés

Couche 4 — Rate Limiting (anti brute-force)
  const loginLimiter = rateLimit({
    windowMs: 15 * 60 * 1000,  // fenêtre de 15 minutes
    max: 10                     // maximum 10 tentatives
  });
  → Bloque après 10 tentatives de connexion échouées

Couche 5 — Mongo Sanitize (anti injection)
  app.use(mongoSanitize());
  → Nettoie les données reçues pour empêcher l'injection MongoDB
  → Exemple bloqué : { "username": { "$gt": "" } }

Couche 6 — bcrypt (mots de passe)
  → Les mots de passe ne sont JAMAIS stockés en clair
  → Même si la base de données est piratée, les mots de passe sont illisibles

Couche 7 — JWT avec expiration
  → Les tokens expirent après 7 jours
  → Secret de signature stocké en variable d'environnement

Couche 8 — Contrôle d'accès par rôle
  function requireRole(...roles) {
    return (req, res, next) => {
      if (!roles.includes(req.user.role)) {
        return res.status(403).json({ error: 'Accès interdit' });
      }
      next();
    };
  }

3.8 LE TEMPS RÉEL (SOCKET.IO)

Socket.IO maintient une connexion permanente entre le client et le serveur.
Quand un événement se produit, le serveur "pousse" l'information vers tous 
les clients concernés sans qu'ils aient besoin de rafraîchir.

Configuration serveur :

  const http = require('http');
  const { Server } = require('socket.io');
  const server = http.createServer(app);
  const io = new Server(server, { cors: { origin: "*" } });

  io.on('connection', (socket) => {
    // Authentifier le client
    const token = socket.handshake.auth.token;
    const user = jwt.verify(token, JWT_SECRET);
    socket.user = user;
    
    // Rejoindre les "rooms" appropriées
    socket.on('join-zone', (zoneId) => {
      socket.join(`zone-${zoneId}`);
    });
  });

Quand un nouveau rapport est créé, le serveur notifie :

  // Dans la route POST /api/reports
  io.emit('new-report', { report: newReport });

Quand un PM laisse un feedback :

  // Notifier uniquement le superviseur concerné
  io.to(`supervisor-${supervisorName}`).emit('new-feedback', { ... });

Système de rooms (salles) :
  - Chaque zone a une room : zone-Zone 1, zone-Zone 2...
  - Chaque superviseur a sa room : supervisor-evariste famba
  - Les messages chat sont envoyés à la room de zone correspondante

3.9 L'UPLOAD D'IMAGES (MULTER + CLOUDINARY)

Deux stratégies de stockage :

Stratégie A — Cloudinary (production) :
  Si les variables CLOUDINARY_* sont définies, les images vont sur le cloud.
  
  const cloudinary = require('cloudinary').v2;
  cloudinary.config({
    cloud_name: process.env.CLOUDINARY_CLOUD_NAME,
    api_key: process.env.CLOUDINARY_API_KEY,
    api_secret: process.env.CLOUDINARY_API_SECRET
  });
  
  const storage = new CloudinaryStorage({
    cloudinary: cloudinary,
    params: { folder: 'daily-report-site-supervisor' }
  });

Stratégie B — Stockage local (développement) :
  Si pas de Cloudinary, les images sont sauvées dans /uploads/
  ⚠️ Perdues à chaque redéploiement sur Render (système éphémère)

Configuration Multer :

  const upload = multer({
    storage: storage,           // Cloudinary ou local
    limits: { fileSize: 10 * 1024 * 1024 },  // Max 10 MB
    fileFilter: (req, file, cb) => {
      const allowed = ['image/jpeg', 'image/png', 'image/heic'];
      if (allowed.includes(file.mimetype)) cb(null, true);
      else cb(new Error('Format non supporté'));
    }
  });

  2. supervisor.js collecte les données des champs
  3. Appel API : fetch('/api/reports', { method: 'POST', body: JSON.stringify(data) })
  4. Si photos sélectionnées : fetch('/api/reports/:id/images', { body: formData })
  5. Serveur stocke en MongoDB + émet Socket.IO 'new-report'
  6. Tous les PM connectés reçoivent le rapport instantanément

7.1 LE SYSTÈME DE ZONES ET MAPPING PROVINCES

Les 26 provinces de la RDC sont réparties en 4 zones.
Le mapping est défini une seule fois dans server.js (objet PROVINCE_TO_ZONE)
et servi aux frontends via l'API /api/zone-mapping.

  function normalizeProvince(str) {
    return String(str).trim().normalize('NFD')
      .replace(/[\u0300-\u036f]/g, '')  // Supprimer les accents
      .toLowerCase();
  }
  
  const PROVINCE_TO_ZONE = {
    [normalizeProvince('Kinshasa')]: 'Zone 1',
    [normalizeProvince('Haut-Katanga')]: 'Zone 4',
    // ... 26 provinces mappées
  };
  
  // Endpoint API (source unique de vérité)
  app.get('/api/zone-mapping', (req, res) => {
    res.json({ success: true, mapping: PROVINCE_TO_ZONE, defaultZone: 'Zone 4' });
  });

Le frontend charge ce mapping au démarrage au lieu d'avoir une copie locale.

7.2 LE SCORING AUTOMATIQUE DES SUPERVISEURS

Formule : Score = (Planning × 60%) + (Qualité × 40%)

  Planning (60%) :
    Si dans les délais → 100%
    Si en retard → max(20, 100 - pourcentage_dépassement)
  
  Qualité (40%) :
    Photos (40%) : 70 + min(30, nb_photos × 2)
    Reporting (35%) : 60 + min(40, nb_rapports × 4)
    Communication (25%) : 60 + min(40, nb_feedbacks × 8)

Chaque phase a un poids (2 à 9) qui influence l'impact de son retard.

PARTIE 8 — LEÇONS APPRISES ET BONNES PRATIQUES
=====================================================

8.1 CE QUI A BIEN MARCHÉ

  ✓ Choix du Vanilla JS : pas de complexité de build, déploiement rapide
  ✓ Monolithique : un seul fichier serveur, facile à comprendre et débugger
  ✓ MongoDB : flexibilité du schéma (ajout de champs sans migration)
  ✓ Socket.IO : UX temps réel appréciée par les utilisateurs
  ✓ Cloudinary : images persistantes sans gestion de stockage
  ✓ Render + GitHub : déploiement en un git push

8.2 LES DÉFIS RENCONTRÉS

  ⚠️ Duplication public/ et public-pm/
    Problème : deux dossiers avec des fichiers presque identiques
    → Risque de désynchronisation
    Solution : script sync-public-pm.js pour synchroniser

  ⚠️ URL serveur en dur dans le frontend
    Problème : getServerUrl() retournait une URL fixe → cassait en local
    Solution : window.location.origin (auto-détection)

  ⚠️ PROVINCE_TO_ZONE triplé
    Problème : le mapping existait dans server.js, pm-dashboard.js, et public-pm
    Solution : API /api/zone-mapping (source unique)

  ⚠️ Mots de passe en dur dans le code
    Problème : mot de passe par défaut écrit dans server.js → visible sur GitHub
    Solution : variable d'environnement SEED_DEFAULT_PASSWORD obligatoire

  ⚠️ Report.find({}) sans filtre
    Problème : l'endpoint site-tracking chargeait TOUS les rapports de la base
    Solution : Report.find({ site_id: { $in: siteIds } }) → filtre par site

  ⚠️ CORS origin: '*'
    Problème : n'importe quel site web pouvait appeler notre API
    Solution : ALLOWED_ORIGINS configurable

8.3 BONNES PRATIQUES APPLIQUÉES

  1. Variables d'environnement pour tous les secrets
  2. Hachage bcrypt pour les mots de passe (jamais en clair)
  3. JWT avec expiration pour l'authentification
  4. Rate limiting contre le brute-force
  5. Validation des données côté serveur
  6. Contrôle d'accès par rôle (RBAC)
  7. mongoSanitize contre l'injection
  8. Git pour le versioning du code
  9. Scripts automatisés (build APK, sync, tests)
  10. Documentation complète (guide utilisateur, glossaire)
